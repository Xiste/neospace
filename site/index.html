Técnicas de Eficiência de Atenção

Flash Attention — Atenção Exata com O(N) de Memória

🟢 Dominado

O Problema

Dados Q, K, V ∈ ℝN×d, a atenção padrão calcula:

Atenção Padrão — 3 passos, 2 escritas N×N na HBM 
S = Q @ K.T          // [N×N] → escreve na HBM
P = softmax(S)        // [N×N] → escreve na HBM
O = P @ V             // [N×d] → escreve na HBM

Total de acessos HBM: Θ(Nd + N²)

Para N=1024, d=64, 16 cabeças, batch=64 em A100: 40.3 GB de leitura/escrita na HBM. O Flash Attention reduz isso para 4.4 GB — 9× menos — eliminando as escritas das matrizes S e P. A GPU não é lenta calculando; é lenta movendo dados.

1,0242 MB10×

Contexto Histórico

Antes de 2022, a comunidade tentava resolver O(N²) com aproximações: sparse attention, Linformer (projeção low-rank), Performer (FAVOR+). Todas perdiam qualidade. O insight veio de Rabe & Staats (2021): "Self-Attention Does Not Need O(n²) Memory" — provaram que tiling funciona para atenção, mas sem implementação IO-aware eficiente. O Flash Attention pegou essa ideia e construiu um kernel CUDA que atinge o limite inferior teórico de acessos à memória.

Intuição Central

A GPU tem dois níveis de memória radicalmente diferentes:

MemóriaCapacidade (A100)BandaLatência
HBM (VRAM)40-80 GB1.5-2.0 TB/s~300 ciclos
SRAM (shared mem)192 KB / SM~19 TB/s~20-30 ciclos

O algoritmo ingênuo trata a GPU como se só existisse HBM. O Flash Attention usa a SRAM como cache explícito: divide as matrizes em tiles que cabem na SRAM, processa cada tile completamente e nunca escreve a matriz N×N na HBM.

📄 Paper 1: FlashAttention (Dao et al., NeurIPS 2022 — Best Paper)

Algoritmo 1 — Forward Pass Completo

Tamanhos de bloco (derivados do tamanho M da SRAM): Bc = ⌈M/(4d)⌉ para K,V; Br = min(⌈M/(4d)⌉, d) para Q,O.

Forward FA1 — Pseudocódigo extraído do paper (Algorithm 1) 
Entrada: Q, K, V ∈ ℝ^(N×d) na HBM, blocos B_c, B_r

1.  Divide Q em T_r = ⌈N/B_r⌉ blocos Q_1...Q_{T_r} de tamanho B_r×d
2.  Divide K, V em T_c = ⌈N/B_c⌉ blocos K_1...K_{T_c}, V_1...V_{T_c}
3.  Inicializa O = (0)_{N×d}, ℓ = (0)_N, m = (-∞)_N na HBM

4.  para j = 1 até T_c:                     // outer loop: K,V blocks
5.      carrega K_j, V_j da HBM → SRAM
6.      para i = 1 até T_r:                 // inner loop: Q blocks
7.          carrega Q_i, O_i, ℓ_i, m_i → SRAM
8.
9.          // Passo 1: scores de atenção
10.         S_ij = τ · Q_i @ K_j^T           // [B_r × B_c] — PEQUENO!
11.         S_ij = MASK(S_ij)                // causal/padding mask
12.
13.         // Passo 2: online softmax
14.         m̃_ij = rowmax(S_ij)              // [B_r,]
15.         P̃_ij = exp(S_ij - m̃_ij)           // [B_r × B_c] pointwise
16.         ℓ̃_ij = rowsum(P̃_ij)              // [B_r,]
17.
18.         // Passo 3: atualiza estatísticas (CORREÇÃO CRÍTICA!)
19.         m_novo = max(m_i, m̃_ij)
20.         ℓ_novo = e^(m_i-m_novo)·ℓ_i + e^(m̃_ij-m_novo)·ℓ̃_ij
21.
22.         // Passo 4: dropout (se aplicável)
23.         P̃_ij = dropout(P̃_ij, p_drop)
24.
25.         // Passo 5: acumula output com correção de escala
26.         O_i = diag(ℓ_novo)⁻¹ · (diag(ℓ_i)·e^(m_i-m_novo)·O_i
27.                                   + e^(m̃_ij-m_novo)·P̃_ij @ V_j)
28.
29.         escreve O_i, ℓ_i←ℓ_novo, m_i←m_novo → HBM
30.     fim para
31. fim para
32. retorna O, ℓ, m, R

O truque do online softmax: a correção e^(m_velho - m_novo) rescala todos os exponenciais acumulados para a base do novo máximo. Quando um bloco novo tem máximo maior que o anterior, os exponenciais antigos precisam ser "encolhidos". Sem isso: overflow se o novo máximo for maior, underflow se for menor. Ver explicação completa no glossário →

Backward Pass — Recomputation

O backward não salva S e P (N×N cada). Recalcula cada bloco S_ij e P_ij na SRAM usando Q_i, K_j e as estatísticas armazenadas.

O que é armazenado (O(N))O que NÃO é armazenado
O (output, N×d)S = QK^T (N×N, ~134 MB p/ N=4096)
ℓ (soma exponenciais, N)P = softmax(S) (N×N, ~134 MB)
m (máximo por linha, N)Dropout mask (regenerada do seed)
R (seed do RNG, O(1))
Backward FA1 — Equações extraídas do paper 
Dado gradiente de saída dO:

1. Gradiente de V:  dV_j = Σ_i P_ij^T @ dO_i

2. Estatística D_i (produto escalar por linha):
   D_i = rowsum(dO_i ∘ O_i)                  // evita reduzir sobre N inteiro!

3. Gradiente via Jacobiano do softmax:
   dS_ij = P_ij ∘ (dP_ij - D_i)             // ∘ = pointwise multiply

4. Gradientes de Q e K:
   dQ_i += τ · dS_ij @ K_j
   dK_j += τ · dS_ij^T @ Q_i

Prova de otimalidade (Proposition 3): Nenhum algoritmo de atenção exata pode ter menos de Ω(N²d²/M) acessos HBM. O Flash Attention atinge esse limite inferior — é assintoticamente ótimo.

📄 Paper 2: FlashAttention-2 (Dao, 2023)

Por que o FA1 era ineficiente (25-40% do pico na A100)

  1. Split-K no particionamento de warps: K e V divididos entre warps, Q compartilhado. Cada warp escrevia seu resultado parcial na shared memory → sincronização → soma. Essas leituras/escritas extras dominavam o tempo.
  2. FLOPs não-matmul excessivos: cada FLOP não-matmul é 16× mais caro que matmul na A100 (312 TFLOPS matmul FP16 vs 19.5 TFLOPS não-matmul FP32).
  3. Baixa ocupação com sequências longas: paralelizava só sobre batch×heads, não sobre sequence length. Com seqlen grande → batch pequeno → GPUs subutilizadas.

Três melhorias do FA2

1. Redução de FLOPs não-matmul:

  • Tweak A: output mantido "não-escalado" até o final do loop. Só normaliza uma vez no fim: O = diag(ℓ)^{-1} · Õ. Elimina uma multiplicação por diag por iteração.
  • Tweak B: em vez de salvar m e ℓ separados, salva L = m + log(ℓ) (logsumexp). Backward usa L diretamente.
Forward FA2 — Pseudocódigo extraído do paper (Algorithm 1) 
Entrada: Q, K, V ∈ ℝ^(N×d) na HBM, blocos B_c, B_r
Saída: O, L (logsumexp)

1.  Divide Q em T_r blocos, K,V em T_c blocos
2.  Divide O em T_r blocos, L em T_r blocos

3.  para i = 1 até T_r EM PARALELO:          // NOVO: paralelo sobre seqlen!
4.      carrega Q_i → SRAM
5.      O_i = (0), ℓ_i = (0), m_i = (-∞)
6.
7.      para j = 1 até T_c:
8.          carrega K_j, V_j → SRAM
9.          S_ij = Q_i @ K_j^T               // [B_r × B_c]
10.
11.         m_novo = max(m_i, rowmax(S_ij))
12.         P̃_ij = exp(S_ij - m_novo)
13.         ℓ_novo = e^(m_i - m_novo)·ℓ_i + rowsum(P̃_ij)
14.
15.         O_i = diag(e^(m_i - m_novo))⁻¹ @ O_i + P̃_ij @ V_j
16.
17.         m_i = m_novo, ℓ_i = ℓ_novo
18.     fim para
19.
20.     O_i = diag(ℓ_i)⁻¹ @ O_i               // normaliza só no final!
21.     L_i = m_i + log(ℓ_i)                  // logsumexp
22.     escreve O_i, L_i → HBM
23. fim para

2. Paralelismo sobre sequence length: Forward paraleliza sobre blocos de linha (Q), backward paraleliza sobre blocos de coluna (K,V). Atualização de dQ usa atomic adds. Aumenta ocupação com batch pequeno.

3. Novo particionamento entre warps (evita split-K):

FA1 (split-K)FA2
QCompartilhado entre warpsDividido entre warps
K, VDivididos entre warpsCompartilhado entre warps
SincronizaçãoWarps escrevem → sync → somamSem comunicação

Cada warp produz sua fatia do output direto, sem precisar sincronizar com outros warps. Elimina leituras/escritas extras na shared memory.

📄 Paper 3: FlashAttention-3 (Dao et al., 2024) — ⚡ Este é o paper para implementar hoje

Por que FA2 atingia só 35% na H100

  1. mma.sync bloqueante: instrução da Ampere que bloqueia a warp. Na H100, só atinge ~2/3 do throughput máximo dos Tensor Cores.
  2. Sem TMA: transferências HBM↔SRAM eram manuais (load/store), consumindo registradores e sem sobreposição com computação.
  3. Sem FP8: throughput FP8 é 2× FP16 (1978 vs 989 TFLOPS na H100 SXM5).
  4. Gargalo da exponencial: unidade de funções especiais tem apenas 3.9 TFLOPS na H100 — 256× menos que matmul FP16. Para head dim 128, exp pode consumir 50% do tempo. Com FP8 seria pior ainda (matmul dobra, exp constante).

Três inovações do FA3

A. WGMMA (Warp Group MMA) + Warp Specialization:

  • WGMMA: instrução assíncrona exclusiva da Hopper. Substitui mma.sync. O warp group (4 warps) dispara a multiplicação e continua executando enquanto ela roda nos Tensor Cores.
  • Inter-Warpgroup Pingpong: dois warpgroups (A: GEMMs, B: softmax) alternam via bar.sync. A softmax roda "na sombra" dos GEMMs do outro warpgroup.
  • Intra-Warpgroup Pipelining: pipeline de 2 estágios com acumuladores separados. Partes do softmax executam enquanto os GEMMs estão em voo. Tradeoff: maior pressão de registradores.

B. TMA (Tensor Memory Accelerator): Hardware dedicado para transferências HBM↔SRAM. Cuida de cálculo de índices e predicação de bounds. Libera registradores. Assíncrono — sobrepõe carga de dados com computação.

C. FP8 com Incoherent Processing:

  • Multiplica Q e K por matriz de Hadamard aleatória — "espalha" outliers para que a energia fique uniforme
  • Custo: O(d log d) por cabeça, fundido com rotary embeddings (praticamente grátis)
  • Erro de quantização 2.6× menor que FP8 baseline
  • Forward inteiro em FP8 (~2× throughput); backward em BF16 para estabilidade

Pré-requisitos

  • Estrutura do Transformer (atenção multi-cabeça, Q/K/V)
  • Hierarquia de memória da GPU (HBM vs SRAM)
  • Modelo de execução CUDA (threads, warps, thread blocks, grid)
  • Online softmax e estabilidade numérica
  • Operações de álgebra linear (GEMM, matmul)
  • Regra da cadeia e backpropagation (para entender o backward)

Conexões

  • Multi-Query / Grouped Query Attention: reduzem cabeças de K,V — FA2 suporta nativamente. Essencial para inferência (reduz KV cache)
  • FSDP: divide parâmetros entre GPUs; combinado com FA permite modelos maiores com contexto maior
  • LoRA: FA acelera forward/backward do modelo base; LoRA reduz parâmetros treináveis
  • Quantização (INT8/FP8): FA3 incorpora FP8 no forward via incoherent processing
  • Sparse Attention: FA1 tem versão block-sparse. Na prática, FA exato é mais usado
  • Flash Decoding: extensão para inferência — paraleliza sobre sequence length do KV cache

🏭 HBM vs SRAM — Hierarquia de Memória da GPU

HBM (High Bandwidth Memory): memória principal da GPU (VRAM). Grande (40-80 GB) mas lenta (latência de centenas de ciclos). Todas as matrizes do modelo ficam aqui. Analogia: um galpão do outro lado da rua — cabe tudo, mas cada viagem é demorada.

SRAM (Static RAM / Shared Memory): memória dentro de cada Streaming Multiprocessor. Pequena (~228 KB por SM na H100) mas muito rápida (~25 ciclos de latência). Analogia: sua mesa de trabalho — cabe pouco, mas tudo ao alcance da mão.

Por que importa no Flash Attention: TODO o algoritmo gira em torno de nunca escrever a matriz N×N na HBM. Em vez disso, processa blocos pequenos na SRAM. Cada acesso evitado à HBM é ganho direto de velocidade.

MemóriaCapacidade (H100)BandaLatência
HBM (VRAM)80 GB3.35 TB/s~300 ciclos
L2 Cache50 MB~12 TB/s~100 ciclos
SRAM (Shared Mem)228 KB / SM~20+ TB/s~25 ciclos
Registradores~64K / SMMáxima~1 ciclo
⬆ Voltar ao Flash Attention

🧩 Sparse Attention

Família de técnicas que calcula apenas um subconjunto das conexões entre tokens, em vez de todas as N×N conexões. É como ler um livro e só consultar palavras do mesmo parágrafo, em vez de comparar cada palavra com todas as outras.

Flash Attention NÃO é sparse attention — ele calcula a atenção exata (todas as conexões). Mas antes de 2022, sparse attention era a principal estratégia para reduzir o custo O(N²) da atenção.

TécnicaEstratégiaQualidade
Sparse Transformer (2019)Padrão fixo: local + stridedPerde conexões distantes
Longformer (2020)Janela deslizante + tokens globaisBoa para NLP longo
BigBird (2020)Aleatório + local + globalMelhor que sparse fixo
Flash Attention (2022)Exata, sem esparsidadeSem perda de qualidade
⬆ Voltar ao Flash Attention

📐 Linformer

Uma das primeiras tentativas de reduzir o custo da atenção usando projeção de baixo rank. A ideia: a matriz de atenção N×N pode ser aproximada por duas matrizes menores N×k e k×N, onde k é pequeno (~256). Complexidade: O(N²) → O(N·k).

Limitação: não escala para sequências mais longas que as do treino. Se treinar com N=512 e inferir com N=4096, degrada.

Relação com Flash Attention: o Linformer mostrou que a matriz de atenção tem baixo rank (muita redundância). Mas é uma aproximação — o Flash Attention veio depois e mostrou que dava para ser exato e eficiente.

⬆ Voltar ao Flash Attention

🎭 Performer

Usa um truque matemático chamado FAVOR+ (Fast Attention Via Orthogonal Random features) para aproximar o softmax com kernels e projeções aleatórias. Complexidade linear O(N) — foi o primeiro mecanismo de atenção linear que funcionou bem.

Performer vs Flash Attention: atacam o mesmo problema de ângulos opostos. Performer muda a matemática (aproximação do softmax → O(N) computação). Flash Attention muda a engenharia (IO-awareness → O(N) memória, computação ainda O(N²)). Para qualidade crítica, FA venceu.

⬆ Voltar ao Flash Attention

🧱 Tiles (Ladrilhamento)

Técnica onde uma matriz grande é dividida em blocos menores que cabem na memória rápida (SRAM). Cada bloco é carregado, processado completamente e descartado antes do próximo.

Analogia: pintar uma parede de 10m×10m — você pinta um quadrado de 1m×1m de cada vez. A parede inteira nunca está na sua mão, só o ladrilho atual.

A matriz de atenção N×N é dividida em tiles de tamanho Br×Bc (ex: 128×128):

Matriz N×N dividida em tiles 
┌─────────┬─────────┬─────────┐
│ Tile    │ Tile    │ Tile    │
│ [0,0]   │ [0,1]   │ [0,2]   │  ← 128×128 cada
├─────────┼─────────┼─────────┤
│ Tile    │ Tile    │ Tile    │
│ [1,0]   │ [1,1]   │ [1,2]   │
├─────────┼─────────┼─────────┤
│ Tile    │ Tile    │ Tile    │
│ [2,0]   │ [2,1]   │ [2,2]   │
└─────────┴─────────┴─────────┘

Cada tile de 128×128 em float16 ocupa 32 KB — cabe na SRAM de 228 KB.

⬆ Voltar ao Flash Attention

⚡ Warp e WGMMA (CUDA)

Warp: grupo de 32 threads que executam juntas a mesma instrução sobre dados diferentes (modelo SIMT). É a unidade fundamental de execução em GPUs NVIDIA. Uma H100 gerencia 64 warps simultaneamente por SM.

WGMMA (Warp Group Matrix Multiply-Accumulate): instrução exclusiva da arquitetura Hopper (H100) que permite que um grupo de warps dispare uma multiplicação de matrizes de forma assíncrona — o warp não espera o resultado, continua executando enquanto a multiplicação roda nos Tensor Cores.

Evolução entre versões do FA:

FA2Cada warp processa um bloco 128×128Menos comunicação entre warps
FA3Warps em estágios diferentes do pipeline (WGMMA)Nenhum warp ocioso

🔢 Online Softmax

Algoritmo que calcula o softmax incrementalmente, processando pedaços do vetor sem nunca ver todos os valores de uma vez. Mantém duas variáveis atualizadas bloco a bloco:

  • m: máximo corrente (para estabilidade numérica)
  • l: soma corrente dos exponenciais (para normalização)

Quando um novo bloco chega, a correção exp(m_velho - m_novo) rescala todos os exponenciais antigos para a nova base. Sem isso, os números estouram (overflow) ou viram zero (underflow).

Exemplo numérico simples 
Bloco 1: scores = [2, 1, 3]
  m = 3, l = e^(2-3) + e^(1-3) + e^(3-3) = 1.503

Bloco 2: scores = [5, 0, 4]  ← novo máximo 5 > 3!
  m_novo = 5
  l = 1.503 * e^(3-5) + e^(5-5) + e^(0-5) + e^(4-5)
    = 1.503 * 0.135 + 1 + 0.007 + 0.368
    = 1.578

O fator e^(3-5) = 0.135 "encolheu" os exponenciais do bloco 1 para a base correta.

Por que importa: é o que viabiliza o tiling no cálculo da atenção. Sem o online softmax, você precisaria da matriz N×N inteira — exatamente o que o Flash Attention evita.
