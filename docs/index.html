1. Tokenização — Byte-Pair Encoding & Subword Tokenization

🟢 Dominado

O Problema

Como representar texto para uma rede neural sem vocabulários gigantes e sem perder palavras raras?

Antes de 2016, a maioria dos modelos de NLP tratava palavras como unidade atômica. Cada palavra recebia um ID fixo: "gato" = 42, "cachorro" = 87, "gatos" = 543. Três problemas graves:

  1. Vocabulário gigante: centenas de milhares de palavras, cada uma ocupando uma linha na matriz de embedding
  2. OOV (Out Of Vocabulary): se "gatinho" não estava no vocabulário de treino, virava <UNK>
  3. Sem relação entre formas parecidas: "gato" e "gatos" eram IDs completamente diferentes. O modelo não sabia que compartilhavam significado

Tentaram character-level (vocabulário ~256) mas as sequências ficavam enormes — e o custo da atenção é O(n²).

Intuição Central — O LEGO das Palavras

Imagine aprender japonês e encontrar 食べ物 (tabemono = comida). Você nunca viu essa palavra. Mas conhece:

  • 食べ (tabe = comer)
  • 物 (mono = coisa)

Se você conhece os pedaços, entende a palavra inteira. O BPE faz exatamente isso: quebra palavras em subpalavras frequentes.

Analogia do LEGO:

  • Word-level = cada palavra é uma peça única e rígida
  • Character-level = cada letra é um pino de LEGO (muito pequeno, difícil de montar)
  • Subword = peças de tamanho certo que se encaixam para formar qualquer palavra
Como o BPE enxerga palavras 
"gatinho"  →  "gat" + "inho"
"gatos"    →  "gat" + "os"
"gato"     →  "gat" + "o"

O radical "gat" carrega o significado felino.
Os sufixos "inho", "os", "o" carregam a gramática.

Como Funciona — Algoritmo Passo a Passo

Inicialização: todo o texto de treino é quebrado em caracteres. Espaços são preservados.

BPE — Passo a passo com "gato gato gatos" 
PASSO 0: Vocabulário inicial = {g, a, t, o, s, _}
         Texto tokenizado: _ g a t o _ g a t o _ g a t o s

PASSO 1: Contar todos os pares adjacentes
         "g a" aparece 3 vezes  →  FUNDIR em "ga"
         Novo vocabulário: {g, a, t, o, s, _, ga}

PASSO 2: "ga t" aparece 3 vezes  →  FUNDIR em "gat"
         Novo vocabulário: {..., gat}

PASSO 3: "gat o" aparece 3 vezes  →  FUNDIR em "gato"

RESULTADO: "gatos" (palavra nova!) → gat o s
           O modelo entende "gatos" mesmo sem nunca ter visto!

O processo repete até atingir o tamanho de vocabulário desejado (~37.000 tokens). As subpalavras mais frequentes emergem naturalmente. Em português, "-mente", "-ção", "des-", "re-" viram tokens próprios.

Na Prática — Números Reais

A tokenização é pré-processamento em CPU — não impacta o tempo de GPU. O que muda na GPU é o tamanho da matriz de embedding: |V| × d_model.

Conexões

  • Embeddings: cada ID de token é convertido em vetor denso de 512 dimensões
  • Weight tying: Transformer compartilha a mesma matriz de embedding na entrada e saída — economiza ~19M parâmetros
  • Flash Attention: a tokenização define n (comprimento da sequência), que determina o tamanho da matriz de atenção O(n²)

2. Embeddings — De Inteiros a Vetores com Significado

🟢 Dominado

O Problema

Como converter IDs de tokens (inteiros) em vetores que capturem significado? Antes dos embeddings densos (pré-2013), usava-se one-hot encoding: um token com ID 542 virava um vetor de 37.000 posições com UM único "1" e o resto zero.

Três problemas catastróficos do one-hot:

  1. Maldição da dimensionalidade: 37.000 dimensões por token, 99.99% zeros — desperdício total de memória
  2. Distância sem sentido: distância("gato", "cachorro") = distância("gato", "abacaxi") = √2. Zero informação semântica
  3. Memória explosiva: um batch de 64 sequências de 512 tokens em one-hot ocuparia 4.8 GB só pra representar palavras

Intuição Central — A Ficha de Biblioteca

Como você descreveria uma pessoa em vez de listar todos os 8 bilhões de humanos?

"Altura média, cabelo escuro, olhos castanhos, usa óculos, fala português..."

5-6 características bastam. Embeddings funcionam igual:

Embeddings: vetores densos que capturam significado 
"rei"     → [0.8, -0.3,  0.5,  0.1, ...]  (512 números)
"rainha"  → [0.6, -0.2,  0.4,  0.1, ...]  (similar!)
"abacaxi" → [-0.5, 0.9, -0.7,  0.3, ...]  (completamente diferente)

Operações famosas:  rei - homem + mulher ≈ rainha

Analogia: One-hot = cada livro tem uma prateleira exclusiva. Embedding = cada livro tem uma ficha com 512 campos: gênero, tema, dificuldade, época... Livros similares têm fichas parecidas.

Matemática — Do Token ao Vetor

A matriz de embedding E tem dimensão |V| × d_model (37.000 × 512). É uma tabela de lookup:

Embedding Lookup 
embedding(token) = E[id_do_token]  // indexação direta, O(1)

Exemplo:
  Token "cat" → ID 8912 → E[8912] → vetor de 512 floats

O fator √d_model: o Transformer multiplica os embeddings por √512 ≈ 22.6. Por quê?

  • Embeddings são inicializados com N(0, 1/d_model) — valores pequenos
  • Positional Encoding usa senos/cossenos entre -1 e 1
  • Sem o fator de escala, o PE dominaria o sinal e o modelo ignoraria o conteúdo semântico no início do treino

Weight Tying: o Transformer usa a MESMA matriz E para embedding do encoder, embedding do decoder e projeção pré-softmax. Economiza 2/3 dos parâmetros de embedding e força consistência: o mesmo espaço vetorial que representa entrada representa saída.

Na Prática — GPU e Memória

O lookup é O(1) por token — indexação direta na GPU, extremamente rápido. A matriz E fica na VRAM o tempo todo durante treino e inferência.

Conexões

  • Positional Encoding: é SOMADO ao embedding para injetar posição
  • Self-Attention: as matrizes W_Q, W_K, W_V recebem os embeddings como entrada
  • Quantização: embeddings ocupam ~58% dos parâmetros do Transformer — alvo prioritário de INT8/INT4

Perguntas de Revisão

3. Self-Attention — O Coração do Transformer

🟢 Dominado

O Problema

Em 2016, o estado da arte em tradução era o modelo seq2seq com atenção de Bahdanau (2014). Mas a atenção era entre codificador e decodificador. Dentro do codificador, as palavras ainda eram processadas sequencialmente por uma RNN:

O gargalo das RNNs 
Frase: "The cat sat on the mat"
RNN:   The → cat → sat → on → the → mat
       Cada palavra só via as anteriores via estado oculto

"the" (posição 0) precisa de "mat" (posição 5)?
→ Informação precisa sobreviver a 5 passos de estado oculto
→ RNNs tinham dificuldade com dependências acima de ~20 tokens

Três problemas: caminho longo O(n) entre tokens distantes, gradiente evanescente (informação se degrada a cada passo), e paralelismo impossível (cada passo depende do anterior).

A ideia revolucionária: e se cada palavra pudesse olhar diretamente para TODAS as outras ao mesmo tempo, em uma única operação?

Intuição Central — A Sala de Aula

Imagine 30 alunos discutindo um texto. Cada aluno tem 3 papéis:

PapelSignificadoPergunta que responde
Query (Q)O que eu quero saber?"Quem tem informação sobre verbos?"
Key (K)O que eu ofereço?"Eu tenho informação sobre sujeitos!"
Value (V)O que eu compartilho?A informação de fato

Como funciona a dinâmica: cada aluno formula sua Query, anuncia sua Key, e compara com todos. Quanto mais compatível Query-Key, mais peso aquele Value recebe na média final.

Exemplo concreto: resolvendo ambiguidade 
"The cat sat on the mat because it was comfortable."

"it" se refere a "mat" ou a "cat"?

Query de "it":     "Quem são meus antecedentes inanimados?"
Key de "mat":      "Sou substantivo inanimado!"  → compatível ✓
Key de "cat":      "Sou animal"                 → compatível ?
Key de "sat":      "Sou ação"                   → incompatível ✗

Resultado: "it" presta 80% de atenção em "mat", 15% em "cat"

Matemática — A Equação Mais Importante do Transformer

Passo 1: Gerar Q, K, V — cada token x_i (512 dims) é projetado por 3 matrizes aprendidas:

Projeções Q, K, V 
Q = X · W_Q     (W_Q: 512 × 64)
K = X · W_K     (W_K: 512 × 64)
V = X · W_V     (W_V: 512 × 64)

d_k = d_model / h = 512 / 8 = 64

Passo 2: Calcular scores — produto escalar de cada Query com TODAS as Keys:

Matriz de Scores (n × n) 
Scores = Q · Kᵀ     (n × 64) · (64 × n) = (n × n)

       K₁   K₂   K₃   K₄
Q₁  [ 0.8  0.1  0.3  0.0 ]  ← token 1 presta mais atenção em 1 e 3
Q₂  [ 0.2  0.9  0.1  0.4 ]
Q₃  [ 0.5  0.1  0.7  0.2 ]
Q₄  [ 0.0  0.3  0.1  0.9 ]

Passo 3: Escalar por √d_k — divide por √64 = 8. Por quê?

Produto escalar de vetores aleatórios de 64 dimensões tem variância = 64. Sem escala, scores podem ser muito grandes (±50), o softmax satura, e o gradiente some. Dividir por √d_k normaliza a variância para ~1:

Efeito do fator de escala 
Sem escala: softmax([50, 5, 3])  = [1.0, 0.0, 0.0]  ← muito concentrado
Com escala: softmax([6.25, 0.63, 0.38]) = [0.85, 0.08, 0.07]  ← saudável

Passo 4: Softmax — normaliza cada linha para soma = 1 (distribuição de probabilidade)

Passo 5: Multiplicar por V — cada token de saída é uma soma ponderada dos Values de TODOS os tokens:

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

Complexidade: $O(n^2 \cdot d)$ — dominada pelo produto $QK^T$

Na Prática — Custo Real na GPU

O produto Q·Kᵀ é ~268M operações para n=512 com d_k=64. Para 6 camadas × 8 cabeças, isso é ~13 GFLOPs só na atenção. É rápido (<10ms em GPU moderna), mas a memória O(n²) é o verdadeiro gargalo — resolvido pelo Flash Attention.

Conexões

  • Self-Attention vs Cross-Attention: na self-attention, Q, K, V vêm da mesma fonte. Na cross-attention, Q vem do decoder e K,V do encoder
  • Flash Attention: resolve exatamente o problema O(n²) em memória usando tiling e recomputação
  • CNN vs RNN vs Self-Attention: caminho máximo O(1) na atenção vs O(n) na RNN vs O(log n) na CNN

4. Multi-Head Attention — 8 Atenções em Paralelo

🟢 Dominado

O Problema

A linguagem tem múltiplos níveis de relação simultâneos. Na frase "Maria disse a João que ela estava cansada":

Tipo de relaçãoPergunta
SintáticaQuem é o sujeito de "estava"?
Correferência"ela" = Maria ou João?
Semântica"cansada" está ligado a "disse" (explicação)?
PosicionalQue palavras estão adjacentes?

Uma única cabeça de atenção precisa representar TUDO isso nos mesmos 64 números. É como forçar um único especialista a ser simultaneamente sintaticista, semanticista e analista de discurso — ele faz uma média e perde nuances. O paper confirma: atenção de cabeça única perde 0.9 BLEU (Tabela 3, linha A).

Intuição Central — Os 8 Detetives Especializados

Você está analisando uma cena de crime. Não chama UM perito. Chama:

  • Perito em digitais → analisa impressões
  • Perito em balística → analisa trajetória
  • Perito em DNA → analisa material genético
  • Perito em documentos → analisa caligrafia

Cada perito olha a MESMA cena com ferramentas diferentes. Depois, todos os laudos são combinados. Multi-Head Attention é isso: 8 cabeças analisando a mesma sequência, cada uma com suas próprias matrizes W_Q, W_K, W_V.

O que cada cabeça aprende (exemplos reais do paper) 
Cabeça 1: "Quem é sujeito de quem?"        → sintaxe
Cabeça 2: "Quais palavras são sinônimas?"   → semântica
Cabeça 3: "O que está perto do quê?"        → posição relativa
Cabeça 4: "Quem se refere a quem?"          → correferência
...
Cabeça 8: (outro padrão emergente)

Na frase "The Law will never be perfect, but its application should be just", uma cabeça mostrou que "its" atendia fortemente a "The Law". O modelo aprendeu correferência sozinho, sem supervisão explícita!

Matemática — 8 Atenções em Paralelo

Cada cabeça i tem suas próprias projeções W_Q_i, W_K_i, W_V_i (512→64). Depois, concatena os 8 resultados e projeta com W_O:

$$\text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h)\,W^O$$ $$\text{head}_i = \text{Attention}(QW^Q_i, KW^K_i, VW^V_i)$$

$h = 8$ cabeças  |  $d_k = d_v = d_{model}/h = 512/8 = 64$
$W^Q_i, W^K_i \in \mathbb{R}^{512 \times 64}$  |  $W^V_i \in \mathbb{R}^{512 \times 64}$  |  $W^O \in \mathbb{R}^{512 \times 512}$
Total: ~1.05M parâmetros

Por que o custo é quase igual ao de 1 cabeça?

Single head ($d_k=512$): $O(n^2 \cdot 512)$
Multi head ($d_k=64$): $8 \times O(n^2 \cdot 64) = O(n^2 \cdot 512)$
O número total de multiplicações é idêntico. A diferença é que 8 cabeças permitem especialização.

Por que h=8 é o ponto ótimo? Muitas cabeças → cada uma tem d_k muito pequeno → perde capacidade de representação. Poucas cabeças → perde especialização. O paper testou e 8 foi o sweet spot empírico.

Na Prática — Visualização e Dropout

As 8 cabeças são processadas como 8 "amostras" em um batch, permitindo paralelização total via batch matrix multiplication na GPU. Dropout de 0.1 é aplicado nos pesos de atenção (após softmax) para evitar dependência excessiva de uma única cabeça.

Conexões

  • Masked Multi-Head Attention: no decoder, a mesma estrutura mas com máscara triangular
  • Cross-Attention: Q vem do decoder, K e V do encoder — mas ainda multi-head
  • Multi-Query Attention (MQA): compartilha K e V entre todas as cabeças para reduzir KV-cache na inferência
  • Grouped Query Attention (GQA): meio-termo: agrupa cabeças que compartilham K,V

Perguntas de Revisão

  1. Por que usar 8 cabeças em vez de 1? O que cada cabeça pode capturar de diferente?
  2. Explique por que 8 × O(n² × 64) = O(n² × 512). O custo é realmente igual?
  3. O que a matriz W_O faz e por que ela é necessária após a concatenação?
  4. Se você inspecionasse uma cabeça e ela sempre produzisse atenção uniforme, o que isso indicaria?
  5. O paper mostra que cabeça única perde 0.9 BLEU, mas cabeças demais também pioram. Qual o trade-off?

Exercício: Se d_model = 1024, quantas cabeças você usaria? Qual seria d_k? Justifique com base no trade-off entre número de cabeças e dimensionalidade.

5. Os 3 Tipos de Atenção no Transformer

🟢 Dominado

O Problema

Modelos seq2seq pré-Transformer comprimiam toda a frase de entrada em um único vetor de contexto. Para frases longas, informação se perdia. A atenção de Bahdanau resolveu parcialmente — o decoder podia visitar cada posição do encoder — mas o encoder em si ainda era sequencial (RNN).

O Transformer unificou tudo sob o mesmo mecanismo de atenção, mas precisava de 3 configurações diferentes para funções diferentes.

Intuição Central — O Tradutor Humano

Pense em um tradutor traduzindo do inglês para português:

1. Entender a frase original (Encoder Self-Attention): lê "The cat that ate the fish is full". Para entender, conecta "is" com "The cat" (e não com "the fish"). Olha para TODAS as palavras livremente.

2. Escrever a tradução em ordem (Decoder Masked Self-Attention): começa a escrever "O gato que comeu o peixe está...". Quando vai escrever "cheio", só pode olhar o que já escreveu. Não pode ver o futuro.

3. Consultar o original (Cross-Attention): ao escrever "cheio", revisita o original: "full" → "cheio", e verifica "is full", "The cat" para garantir concordância.

Os 3 Tipos — Comparação Visual

#TipoQ vem deK, V vêm deMáscara?Custo

Tipo 1: Encoder Self-Attention (Sem Máscara)

Todos os tokens podem ver todos os outros. A matriz de atenção é completa:

Matriz de atenção do encoder (completa) 
       The  cat  sat  on  mat
The  [ 0.3  0.2  0.1  0.2  0.2 ]  ← "The" pode ver "mat"!
cat  [ 0.1  0.4  0.2  0.2  0.1 ]
sat  [ 0.1  0.1  0.5  0.2  0.1 ]
on   [ 0.1  0.1  0.1  0.5  0.2 ]
mat  [ 0.0  0.1  0.1  0.3  0.5 ]

Todas as células são preenchidas. Cada token contextualiza
seu significado com a frase inteira de uma só vez.

Tipo 2: Decoder Masked Self-Attention

ANTES do softmax, aplica-se uma máscara com $-\infty$ nas posições futuras. O softmax de $-\infty = 0$:

$$M_{ij} = \begin{cases} 0 & \text{se } j \leq i \\ -\infty & \text{se } j > i \end{cases}$$ $$\text{softmax}(QK^T + M) = \begin{bmatrix} 1.0 & 0 & 0 & 0 \\ 0.3 & 0.7 & 0 & 0 \\ 0.1 & 0.2 & 0.7 & 0 \\ 0.1 & 0.1 & 0.3 & 0.5 \end{bmatrix}$$

Por que isso é necessário? Durante o treino, o modelo recebe a sequência completa (teacher forcing). A máscara impede que ele "cole" olhando a resposta correta na posição seguinte.

Tipo 3: Cross-Attention (Encoder→Decoder)

O decoder precisa de informação da frase de entrada. Q vem do decoder ("o que estou tentando gerar?"), K e V vêm do encoder ("aqui está o significado da entrada"):

6. Positional Encoding — Como o Modelo Sabe a Ordem?

🟢 Dominado

O Problema

RNNs processam tokens em ordem — a posição é implícita na arquitetura. CNNs têm noção de posição via janela de convolução. O Transformer removeu AMBAS.

A self-attention é uma operação de conjunto: trata a sequência como um conjunto não-ordenado. Para a atenção, "The cat sat on the mat" e "mat the on sat cat The" produziriam a MESMA representação. Sem informação de posição, "A ama B" = "B ama A".

Intuição Central — O Relógio de 256 Ponteiros

Como codificar "compasso 42" para 100 cantores de um coral? Abordagens possíveis:

  • Inteiro bruto "42!": o número 42 não tem relação natural com 41 ou 43. O modelo teria que aprender do zero que posições adjacentes são relacionadas
  • Vetor aprendido por posição: funciona mas não generaliza para sequências mais longas que as do treino
  • Código binário contínuo (Transformer): use ondas de diferentes frequências!

Analogia do relógio:

  • Ponteiro dos segundos → completa uma volta a cada 60s (alta frequência)
  • Ponteiro dos minutos → completa uma volta a cada 3600s (média)
  • Ponteiro das horas → completa uma volta a cada 43200s (baixa)

A combinação única das posições dos 3 ponteiros codifica qualquer horário do dia com precisão de segundos. O Positional Encoding faz o mesmo com 256 "ponteiros" (pares seno/cosseno) em vez de 3.

Matemática — Senos e Cossenos

$$PE_{(pos,\,2i)} = \sin\!\left(\frac{pos}{10000^{2i/d_{model}}}\right)$$ $$PE_{(pos,\,2i+1)} = \cos\!\left(\frac{pos}{10000^{2i/d_{model}}}\right)$$

$pos = 0, 1, 2, \ldots, n-1$  |  $i = 0, 1, \ldots, d_{model}/2 - 1$  |  $d_{model} = 512$

As frequências formam progressão geométrica:

i (par)Frequência = 10000^(2i/512)PeríodoComportamento
010000⁰ = 1~2πOscila muito rápido
12810000^0.5 = 100~200πOscilação média
25510000^0.996 ≈ 9772~19544πOscila muito devagar

Dimensões baixas (i pequeno) oscilam rápido — capturam posições vizinhas. Dimensões altas (i grande) oscilam devagar — capturam a região geral do token na sequência.

Por que seno E cosseno especificamente? Porque a trigonometria tem uma propriedade especial: `sen(a+b)` e `cos(a+b)` podem ser reescritos como combinações lineares de `sen(a)`, `cos(a)`, `sen(b)` e `cos(b)`. Isso não vale para tangente, sigmoid ou outras funções.

Essa propriedade permite que o modelo aprenda posições relativas ("estou a 3 tokens de distância de X"), não apenas absolutas ("sou o token 7"). É a razão matemática mais profunda do design senoidal.

Por que "PE(pos+k) é linear em PE(pos)" é a Chave de Tudo

Lembra das identidades trigonométricas?

Identidades do ensino médio que fazem o PE funcionar
$$\sin(a + b) = \sin(a)\cos(b) + \cos(a)\sin(b)$$ $$\cos(a + b) = \cos(a)\cos(b) - \sin(a)\sin(b)$$

Agora aplica isso ao PE na posição $pos + k$ para uma dimensão par $2i$:

$$PE_{(pos+k,\,2i)} = \sin\!\left(\frac{pos + k}{10000^{2i/d}}\right)$$ $$= \sin\!\left(\frac{pos}{10000^{2i/d}} + \frac{k}{10000^{2i/d}}\right)$$ $$= \underbrace{\sin\!\left(\frac{pos}{10000^{2i/d}}\right)}_{PE_{(pos,2i)}} \cdot \cos\!\left(\frac{k}{10000^{2i/d}}\right) + \underbrace{\cos\!\left(\frac{pos}{10000^{2i/d}}\right)}_{PE_{(pos,2i+1)}} \cdot \sin\!\left(\frac{k}{10000^{2i/d}}\right)$$ $$= PE_{(pos,\,2i)} \cdot A + PE_{(pos,\,2i+1)} \cdot B$$

$A = \cos(k/10000^{2i/d})$ e $B = \sin(k/10000^{2i/d})$ dependem APENAS de $k$ (distância), não de $pos$!

Traduzindo para a prática: PE(pos+k) é uma combinação linear de PE(pos). Os coeficientes dependem apenas de k (distância), não de pos (posição absoluta). Isso significa que se o modelo aprende "atenção ao token que está a 3 posições de distância", essa mesma transformação funciona em QUALQUER lugar da sequência — pos=5, pos=100, ou pos=5000.

Analogia final: uma régua normal tem números absolutos (1, 2, 3...) — cada marcação é independente. Com PE senoidal, o "código" de cada posição é construído de forma que a distância entre duas posições é codificada diretamente na relação linear entre seus vetores. É como uma régua onde o padrão se repete, e o deslocamento de "+3 casas" é sempre a mesma operação matemática.

Como é Usado — Simples Assim

Embedding + Positional Encoding 
1. Token "cat" na posição 3 → Embedding: vetor de 512 floats
2. Calcular PE para posição 3 → PE(3): vetor de 512 floats
3. SOMAR: entrada_da_atenção = Embedding + PE(3)

Por que SOMAR e não CONCATENAR?
  Somar preserva d_model = 512 (concat dobraria para 1024).
  O modelo aprende a separar sinal semântico do posicional.
  Embeddings × √512 equilibra a magnitude com PE (±1).

Na Prática — Senoidal vs Aprendido

O paper testou as duas opções (Tabela 3, linha E): resultados quase idênticos. A versão senoidal foi escolhida pela esperança de extrapolar para sequências mais longas. Na prática moderna, modelos como GPT e LLaMA usam PE aprendido ou RoPE (Rotary Position Embedding) — que aplica rotação nos Q e K em vez de somar.

Conexões

  • Embeddings × √d_model: sem esse fator, o PE (±1) dominaria o sinal semântico no início do treino
  • RoPE (Rotary Position Embedding): evolução usada em LLaMA e Mistral — codifica posição como rotação nos vetores Q e K, não como adição
  • ALiBi (Attention with Linear Biases): alternativa sem PE explícito — adiciona um viés linear negativo à matriz de atenção. Simples e eficaz
  • Contexto longo: PE senoidal permite teoricamente extrapolar para sequências maiores que as do treino — RoPE também com ajuste de frequência

7. Feed-Forward, Conexões Residuais e LayerNorm

🟢 Dominado

O Problema

Redes profundas (20+ camadas) eram impossíveis de treinar até 2015. Dois problemas:

  1. Gradiente evanescente: o sinal de erro ficava cada vez menor a cada camada retropropagada. As primeiras camadas não aprendiam nada
  2. Degradação: mesmo com gradientes OK, redes mais profundas tinham erro de treino MAIOR que redes rasas. Adicionar camadas piorava o modelo

O Transformer empilha 6+6=12 camadas. Sem as técnicas certas, seria impossível treinar.

Intuição Central — Três Peças, Três Analogias

Feed-Forward Network (FFN) — "O processamento individual": A atenção misturou informação entre tokens (reunião de equipe). Agora cada token precisa processar sozinho o que aprendeu. A FFN é aplicada identicamente a cada token, mas de forma independente:

FFN: mesma transformação para cada token 
Token 1 → FFN → Token 1'  (não olha para outros tokens)
Token 2 → FFN → Token 2'
Token 3 → FFN → Token 3'

Expande 512 → 2048 (rascunho em folha grande)
Contrai 2048 → 512 (resumo em post-it)

Conexão Residual — "O atalho da memória": Em vez de reescrever o documento do zero, faça pequenas edições no texto existente:

Residual: edite, não reescreva 
Em vez de:  saída = Atenção(x)
Usamos:     saída = Atenção(x) + x    ← "atalho"

Se a camada não tiver nada útil a acrescentar,
pode zerar sua saída e o sinal original passa intacto.
O gradiente sempre tem um caminho direto de volta.

LayerNorm — "O equalizador de áudio": Após somar Atenção(x) + x, os valores podem ter magnitudes muito diferentes. O LayerNorm ajusta cada token para média 0 e desvio padrão 1:

LayerNorm: normalize por token 
Para cada token (512 dimensões):
  média = soma(dimensões) / 512
  desvio = sqrt(soma((dim - média)²) / 512)
  normalizado = (dim - média) / desvio

Depois: γ × normalizado + β  (γ e β são aprendidos)

Matemática

Feed-Forward Network completa:

$$\text{FFN}(x) = \text{ReLU}(xW_1 + b_1)W_2 + b_2$$

$W_1 \in \mathbb{R}^{512 \times 2048}$ (expansão $4\times$)  |  $W_2 \in \mathbb{R}^{2048 \times 512}$ (contração)

LayerNorm:

$$\mu = \frac{1}{d_{model}}\sum_i x_i \qquad \sigma = \sqrt{\frac{1}{d_{model}}\sum_i (x_i - \mu)^2}$$ $$\text{LayerNorm}(x) = \gamma \cdot \frac{x - \mu}{\sigma} + \beta$$

$\gamma, \beta \in \mathbb{R}^{d_{model}}$ — aprendidos, permitem "desfazer" a normalização

BatchNorm vs LayerNorm — diferença crucial:

BatchNormLayerNorm
Normaliza...Mesma dimensão, todos os tokens do batchTodas as dimensões, cada token individualmente
Depende do batch?Sim — ruim para batches pequenosNão — perfeito para sequências
InferênciaPrecisa de estatísticas móveisIdêntico ao treino

Ordem Exata em Cada Sub-Camada

Pós-norm (Transformer original) 
x entra
  → SubLayer(x)      (atenção ou FFN)
  → Dropout(0.1)
  → + x              (conexão residual — o "atalho")
  → LayerNorm        (equalizador)
  → saída

Esta ordem se chama "pós-norm" (LayerNorm DEPOIS da residual).
Modelos modernos (GPT, LLaMA) usam "pré-norm" (LayerNorm ANTES).

Conexões

  • Pré-norm vs Pós-norm: modelos modernos preferem pré-norm (LayerNorm ANTES da SubLayer) — treino mais estável, permite taxa de aprendizado maior sem warmup
  • SwiGLU: substitui ReLU na FFN por uma função mais expressiva — usada em LLaMA, PaLM

Treinamento — O Loop Completo

Otimizador Adam com Warmup:

$$lr = d_{model}^{-0.5} \cdot \min\!\big(step\_num^{-0.5},\; step\_num \cdot warmup\_steps^{-1.5}\big)$$

Passos $0 \to 4000$: lr aumenta linearmente (warmup)
Passos $4000+$: lr decai $\propto 1/\sqrt{step}$
Adam: $\beta_1=0.9,\; \beta_2=0.98,\; \varepsilon=10^{-9}$  |  Batch: ~25K tokens

Por que warmup? No início, os pesos são aleatórios e a atenção pode produzir sinais instáveis. Começar com lr muito alto faria o modelo divergir. O warmup aumenta gradualmente, permitindo que o otimizador se estabilize.

Regularizações:

  • Dropout (p=0.1): zera aleatoriamente 10% das ativações em cada forward pass. Funciona como um "ensemble implícito" de sub-redes
  • Label Smoothing (ε=0.1): em vez de treinar com "100% certeza neste token", usa 90% no token certo e 10% distribuído entre os outros 37K tokens. Piora perplexidade (modelo menos "certo"), mas melhora BLEU (traduções melhores)

Inferência — Gerando Texto

Beam Search (tamanho 4): em vez de escolher o token mais provável a cada passo (greedy), mantém 4 hipóteses paralelas. A cada passo, expande cada hipótese e mantém as 4 melhores sequências.

Penalidade de comprimento (α=0.6): sem isso, o modelo favorece frases muito curtas (cada token adicional multiplica mais probabilidades). A penalidade ajusta favorecendo cobertura adequada.

Técnicas de Eficiência de Atenção

Flash Attention — Atenção Exata com O(N) de Memória

🟢 Dominado

O Problema

Dados $Q, K, V \in \mathbb{R}^{N \times d}$, a atenção padrão calcula:

$$S = QK^T \in \mathbb{R}^{N \times N}, \quad P = \text{softmax}(S), \quad O = PV \in \mathbb{R}^{N \times d}$$

Cada passo escreve e lê da HBM. Total de acessos: $\Theta(Nd + N^2)$

Para $N=1024$, $d=64$, 16 cabeças, batch=64 em A100: 40.3 GB de leitura/escrita na HBM. O Flash Attention reduz isso para 4.4 GB — 9× menos — eliminando as escritas das matrizes $S$ e $P$. A GPU não é lenta calculando; é lenta movendo dados.

Contexto Histórico

Antes de 2022, a comunidade tentava resolver O(N²) com aproximações: sparse attention, Linformer (projeção low-rank), Performer (FAVOR+). Todas perdiam qualidade. O insight veio de Rabe & Staats (2021): "Self-Attention Does Not Need O(n²) Memory" — provaram que tiling funciona para atenção, mas sem implementação IO-aware eficiente. O Flash Attention pegou essa ideia e construiu um kernel CUDA que atinge o limite inferior teórico de acessos à memória.

Intuição Central

A GPU tem dois níveis de memória radicalmente diferentes:

MemóriaCapacidade (A100)BandaLatência
HBM (VRAM)40-80 GB1.5-2.0 TB/s~300 ciclos
SRAM (shared mem)192 KB / SM~19 TB/s~20-30 ciclos

O algoritmo ingênuo trata a GPU como se só existisse HBM. O Flash Attention usa a SRAM como cache explícito: divide as matrizes em tiles que cabem na SRAM, processa cada tile completamente e nunca escreve a matriz N×N na HBM.

Algoritmo 1 — Forward Pass Completo

Tamanhos de bloco (derivados do tamanho M da SRAM): Bc = ⌈M/(4d)⌉ para K,V; Br = min(⌈M/(4d)⌉, d) para Q,O.

Forward FA1 — Pseudocódigo extraído do paper (Algorithm 1) 
Entrada: Q, K, V ∈ ℝ^(N×d) na HBM, blocos B_c, B_r

1.  Divide Q em T_r = ⌈N/B_r⌉ blocos Q_1...Q_{T_r} de tamanho B_r×d
2.  Divide K, V em T_c = ⌈N/B_c⌉ blocos K_1...K_{T_c}, V_1...V_{T_c}
3.  Inicializa O = (0)_{N×d}, ℓ = (0)_N, m = (-∞)_N na HBM

4.  para j = 1 até T_c:                     // outer loop: K,V blocks
5.      carrega K_j, V_j da HBM → SRAM
6.      para i = 1 até T_r:                 // inner loop: Q blocks
7.          carrega Q_i, O_i, ℓ_i, m_i → SRAM
8.
9.          // Passo 1: scores de atenção
10.         S_ij = τ · Q_i @ K_j^T           // [B_r × B_c] — PEQUENO!
11.         S_ij = MASK(S_ij)                // causal/padding mask
12.
13.         // Passo 2: online softmax
14.         m̃_ij = rowmax(S_ij)              // [B_r,]
15.         P̃_ij = exp(S_ij - m̃_ij)           // [B_r × B_c] pointwise
16.         ℓ̃_ij = rowsum(P̃_ij)              // [B_r,]
17.
18.         // Passo 3: atualiza estatísticas (CORREÇÃO CRÍTICA!)
19.         m_novo = max(m_i, m̃_ij)
20.         ℓ_novo = e^(m_i-m_novo)·ℓ_i + e^(m̃_ij-m_novo)·ℓ̃_ij
21.
22.         // Passo 4: dropout (se aplicável)
23.         P̃_ij = dropout(P̃_ij, p_drop)

27.                                   + e^(m̃_ij-m_novo)·P̃_ij @ V_j)

O truque do online softmax: a correção e^(m_velho - m_novo) rescala todos os exponenciais acumulados para a base do novo máximo. Quando um bloco novo tem máximo maior que o anterior, os exponenciais antigos precisam ser "encolhidos". Sem isso: overflow se o novo máximo for maior, underflow se for menor. O backward não salva S e P (N×N cada). Recalcula cada bloco S_ij e P_ij na SRAM usando Q_i, K_j e as estatísticas armazenadas.

Dado gradiente de saída dO:

1. Gradiente de V:  dV_j = Σ_i P_ij^T @ dO_i

2. Estatística D_i (produto escalar por linha):
   D_i = rowsum(dO_i ∘ O_i)                  // evita reduzir sobre N inteiro!

3. Gradiente via Jacobiano do softmax:
   dS_ij = P_ij ∘ (dP_ij - D_i)             // ∘ = pointwise multiply

📄 Paper 2: FlashAttention-2 (Dao, 2023)

Por que o FA1 era ineficiente (25-40% do pico na A100)

  1. Split-K no particionamento de warps: K e V divididos entre warps, Q compartilhado. Cada warp escrevia seu resultado parcial na shared memory → sincronização → soma. Essas leituras/escritas extras dominavam o tempo.
  2. FLOPs não-matmul excessivos: cada FLOP não-matmul é 16× mais caro que matmul na A100 (312 TFLOPS matmul FP16 vs 19.5 TFLOPS não-matmul FP32).
  3. Baixa ocupação com sequências longas: paralelizava só sobre batch×heads, não sobre sequence length. Com seqlen grande → batch pequeno → GPUs subutilizadas.

Três melhorias do FA2

1. Redução de FLOPs não-matmul:

  • Tweak A: output mantido "não-escalado" até o final do loop. Só normaliza uma vez no fim: O = diag(ℓ)^{-1} · Õ. Elimina uma multiplicação por diag por iteração.
  • Tweak B: em vez de salvar m e ℓ separados, salva L = m + log(ℓ) (logsumexp). Backward usa L diretamente.

📄 Paper 3: FlashAttention-3 (Dao et al., 2024) — ⚡ Este é o paper para implementar hoje

Por que FA2 atingia só 35% na H100

  1. mma.sync bloqueante: instrução da Ampere que bloqueia a warp. Na H100, só atinge ~2/3 do throughput máximo dos Tensor Cores.
  2. Sem TMA: transferências HBM↔SRAM eram manuais (load/store), consumindo registradores e sem sobreposição com computação.
  3. Sem FP8: throughput FP8 é 2× FP16 (1978 vs 989 TFLOPS na H100 SXM5).
  4. Gargalo da exponencial: unidade de funções especiais tem apenas 3.9 TFLOPS na H100 — 256× menos que matmul FP16. Para head dim 128, exp pode consumir 50% do tempo. Com FP8 seria pior ainda (matmul dobra, exp constante).

Três inovações do FA3

A. WGMMA (Warp Group MMA) + Warp Specialization:

  • WGMMA: instrução assíncrona exclusiva da Hopper. Substitui mma.sync. O warp group (4 warps) dispara a multiplicação e continua executando enquanto ela roda nos Tensor Cores.
  • Inter-Warpgroup Pingpong: dois warpgroups (A: GEMMs, B: softmax) alternam via bar.sync. A softmax roda "na sombra" dos GEMMs do outro warpgroup.
  • Intra-Warpgroup Pipelining: pipeline de 2 estágios com acumuladores separados. Partes do softmax executam enquanto os GEMMs estão em voo. Tradeoff: maior pressão de registradores.

B. TMA (Tensor Memory Accelerator): Hardware dedicado para transferências HBM↔SRAM. Cuida de cálculo de índices e predicação de bounds. Libera registradores. Assíncrono — sobrepõe carga de dados com computação.

C. FP8 com Incoherent Processing:

  • Multiplica Q e K por matriz de Hadamard aleatória — "espalha" outliers para que a energia fique uniforme
  • Custo: O(d log d) por cabeça, fundido com rotary embeddings (praticamente grátis)
  • Erro de quantização 2.6× menor que FP8 baseline
  • Forward inteiro em FP8 (~2× throughput); backward em BF16 para estabilidade

🏭 HBM vs SRAM — Hierarquia de Memória da GPU

HBM (High Bandwidth Memory): memória principal da GPU (VRAM). Grande (40-80 GB) mas lenta (latência de centenas de ciclos). Todas as matrizes do modelo ficam aqui. Analogia: um galpão do outro lado da rua — cabe tudo, mas cada viagem é demorada.

SRAM (Static RAM / Shared Memory): memória dentro de cada Streaming Multiprocessor. Pequena (~228 KB por SM na H100) mas muito rápida (~25 ciclos de latência). Analogia: sua mesa de trabalho — cabe pouco, mas tudo ao alcance da mão.

Por que importa no Flash Attention: TODO o algoritmo gira em torno de nunca escrever a matriz N×N na HBM. Em vez disso, processa blocos pequenos na SRAM. Cada acesso evitado à HBM é ganho direto de velocidade.

🧩 Sparse Attention

Família de técnicas que calcula apenas um subconjunto das conexões entre tokens, em vez de todas as N×N conexões. É como ler um livro e só consultar palavras do mesmo parágrafo, em vez de comparar cada palavra com todas as outras.

Flash Attention NÃO é sparse attention — ele calcula a atenção exata (todas as conexões). Mas antes de 2022, sparse attention era a principal estratégia para reduzir o custo O(N²) da atenção.

📐 Linformer

Uma das primeiras tentativas de reduzir o custo da atenção usando projeção de baixo rank. A ideia: a matriz de atenção N×N pode ser aproximada por duas matrizes menores N×k e k×N, onde k é pequeno (~256). Complexidade: O(N²) → O(N·k).

Limitação: não escala para sequências mais longas que as do treino. Se treinar com N=512 e inferir com N=4096, degrada.

Relação com Flash Attention: o Linformer mostrou que a matriz de atenção tem baixo rank (muita redundância). Mas é uma aproximação — o Flash Attention veio depois e mostrou que dava para ser exato e eficiente.

⬆ Voltar ao Flash Attention

🎭 Performer

Usa um truque matemático chamado FAVOR+ (Fast Attention Via Orthogonal Random features) para aproximar o softmax com kernels e projeções aleatórias. Complexidade linear O(N) — foi o primeiro mecanismo de atenção linear que funcionou bem.

Performer vs Flash Attention: atacam o mesmo problema de ângulos opostos. Performer muda a matemática (aproximação do softmax → O(N) computação). Flash Attention muda a engenharia (IO-awareness → O(N) memória, computação ainda O(N²)). Para qualidade crítica, FA venceu.

🧱 Tiles (Ladrilhamento)

Técnica onde uma matriz grande é dividida em blocos menores que cabem na memória rápida (SRAM). Cada bloco é carregado, processado completamente e descartado antes do próximo.

Analogia: pintar uma parede de 10m×10m — você pinta um quadrado de 1m×1m de cada vez. A parede inteira nunca está na sua mão, só o ladrilho atual.

A matriz de atenção N×N é dividida em tiles de tamanho Br×Bc (ex: 128×128):

├─────────┼─────────┼─────────┤
│ Tile    │ Tile    │ Tile    │
│ [1,0]   │ [1,1]   │ [1,2]   │
├─────────┼─────────┼─────────┤

Cada tile de 128×128 em float16 ocupa 32 KB — cabe na SRAM de 228 KB.

⚡ Warp e WGMMA (CUDA)

Warp: grupo de 32 threads que executam juntas a mesma instrução sobre dados diferentes (modelo SIMT). É a unidade fundamental de execução em GPUs NVIDIA. Uma H100 gerencia 64 warps simultaneamente por SM.

WGMMA (Warp Group Matrix Multiply-Accumulate): instrução exclusiva da arquitetura Hopper (H100) que permite que um grupo de warps dispare uma multiplicação de matrizes de forma assíncrona — o warp não espera o resultado, continua executando enquanto a multiplicação roda nos Tensor Cores.

🔢 Online Softmax

Algoritmo que calcula o softmax incrementalmente, processando pedaços do vetor sem nunca ver todos os valores de uma vez. Mantém duas variáveis atualizadas bloco a bloco:

  • m: máximo corrente (para estabilidade numérica)
  • l: soma corrente dos exponenciais (para normalização)

Quando um novo bloco chega, a correção exp(m_velho - m_novo) rescala todos os exponenciais antigos para a nova base. Sem isso, os números estouram (overflow) ou viram zero (underflow).

Exemplo numérico simples
$$\begin{aligned} \text{Bloco 1: scores } &= [2, 1, 3] \\ m &= 3 \\ \ell &= e^{2-3} + e^{1-3} + e^{3-3} = 1.503 \\[8pt] \text{Bloco 2: scores } &= [5, 0, 4] \quad \leftarrow \text{novo máximo } 5 > 3! \\ m_{novo} &= 5 \\ \ell &= 1.503 \cdot e^{3-5} + e^{5-5} + e^{0-5} + e^{4-5} \\ &= 1.503 \cdot 0.135 + 1 + 0.007 + 0.368 \\ &= 1.578 \end{aligned}$$

O fator $e^{3-5} = 0.135$ "encolheu" os exponenciais do bloco 1 para a base correta.

Por que importa: é o que viabiliza o tiling no cálculo da atenção. Sem o online softmax, você precisaria da matriz N×N inteira — exatamente o que o Flash Attention evita.

🔢 Softmax

Função que converte um vetor de números quaisquer em probabilidades que somam 1. É a função mais usada em redes neurais quando o modelo precisa "escolher" entre várias opções.

$$\text{softmax}(z_i) = \frac{e^{z_i}}{\sum_{j=1}^{n} e^{z_j}}$$

Intuição: um concurso onde cada candidato recebe uma nota. A softmax transforma notas em "chance de ser escolhido" — notas maiores ganham mais chance, mas todos recebem pelo menos um pouco. Ex: notas [2, 1, 0] → softmax → [67%, 24%, 9%].

Onde aparece: atenção (converte scores $QK^T$ em pesos), saída do Transformer (logits → probabilidades).

📊 BLEU (Bilingual Evaluation Understudy)

Métrica de 0 a 100 que mede a qualidade de tradução automática comparando a tradução do modelo com traduções humanas de referência. Quanto mais alto, melhor.

Como funciona (intuição): conta quantos pedaços de 1, 2, 3 e 4 palavras (n-gramas) da tradução do modelo também aparecem na referência humana. Penaliza traduções muito curtas.

No Transformer (2017): 28.4 BLEU em EN→DE era o melhor resultado já obtido para tradução inglês→alemão na época.

📐 Notação Big-O — O(n²), O(n), O(1)

Notação matemática que descreve como algo cresce quando o tamanho da entrada (n) aumenta. Não é um número exato — é a tendência.

NotaçãoSignificadoExemplo real
$O(1)$Tempo/memória constanteAcessar array por índice
$O(n)$Cresce linearmenteSomar elementos de uma lista
$O(n^2)$Cresce quadraticamenteAtenção padrão (cada token × cada token)
$O(n \cdot d)$Linear em ambas dimensõesMultiplicar matriz $n \times d$ por vetor

Por que importa: o problema central da atenção é O(n²) em memória — se a sequência dobra, a memória quadruplica. Flash Attention reduz isso para O(n).

📏 d_model, d_k, d_v, d_ff — A Notação de Dimensões

No Transformer, toda dimensão segue a convenção d_algumaCoisa. O d vem de dimension (dimensão):

NotaçãoSignificadoValor (Base)Onde aparece
d_modelDimension of the model — tamanho do embedding e camadas ocultas512Em tudo — embeddings, atenção, FFN, saída
d_kDimension of the Key — tamanho de cada Key por cabeça64 (=512/8)Q·Kᵀ na atenção
d_vDimension of the Value — tamanho de cada Value por cabeça64 (=512/8)Pesos·V na atenção
d_ffDimension of the Feed-Forward — tamanho da FFN oculta2048 (=512×4)Feed-Forward Network
hNúmero de heads (cabeças)8Multi-Head Attention

Relação fundamental: d_k = d_v = d_model / h = 512/8 = 64. As dimensões não são números mágicos — todas derivam de d_model e h.

Analogia da fábrica: uma esteira de 512 cm de largura (d_model). Tudo que passa — embeddings, atenção, FFN — tem 512 cm. A atenção divide em 8 faixas de 64 cm (d_k, d_v). A FFN expande para 2048 cm temporariamente e contrai de volta.

Modelos modernos: GPT-3 usa d_model=12288, LLaMA-7B usa 4096. O valor dobra/escala conforme o modelo cresce.

💾 FP16, BF16, FP32, FP8 — Formatos de Ponto Flutuante

Formatos que definem quantos bits representam cada número e como são distribuídos entre precisão e alcance.

FormatoBitsExpoenteMantissaAlcancePrecisão
FP3232823±3.4×10³⁸Alta (~7 dígitos)

Intuição (régua): FP32 = régua de 1km com marcação a cada mm. BF16 = régua de 1km com marcação a cada 3cm (mesmo alcance, menos precisão). FP8 = régua de 50cm (pouco alcance).

No contexto: Mixed Precision treina com FP16 (rápido) mas mantém cópias críticas em FP32 (estável). FA3 usa FP8 para 2× mais velocidade na H100. BF16 é preferido em modelos modernos (GPT, LLaMA) por ter o mesmo alcance do FP32.

📉 Gradiente e Backpropagation

Gradiente: vetor que aponta a direção de subida mais íngreme de uma função. Em IA, usamos o gradiente da função de perda (erro) para saber como ajustar cada peso do modelo.

Analogia: você está no topo de uma montanha com neblina e quer descer ao vale. O gradiente é como tatear o chão com o pé para saber qual direção desce mais. Cada passo na direção oposta ao gradiente te leva mais para baixo.

Backpropagation: algoritmo que calcula o gradiente para TODOS os pesos do modelo de uma vez, aplicando a regra da cadeia da última camada até a primeira. É o que torna o treinamento de redes profundas possível.

🎯 Cross-Entropy (Entropia Cruzada)

Função de perda (loss) que mede a diferença entre a previsão do modelo e o valor real. É a função padrão para classificação e modelos de linguagem.

Intuição: se o modelo prevê 90% de chance para a palavra correta, cross-entropy ≈ 0.1 (baixa). Se prevê só 10%, ≈ 2.3 (alta). O treino minimiza essa função para o modelo ficar cada vez mais "certo".

$$\mathcal{L} = -\sum_i y_i \log(p_i)$$

$y_i$ é o valor real (1 para classe correta, 0 para as outras) e $p_i$ é a previsão do modelo.

❓ Perplexity (Perplexidade)

Métrica que mede o quão "surpreso" o modelo fica ao ver um texto: perplexity = e^(cross_entropy).

Intuição: perplexidade 10 significa que, em média, o modelo hesita entre 10 palavras possíveis a cada posição. Perplexidade 1 = certeza absoluta (nunca acontece com linguagem real). Quanto menor, melhor.

🚀 GEMM (General Matrix Multiply)

Operação de multiplicação de matrizes que as GPUs executam em hardware dedicado (Tensor Cores). É a operação mais importante em deep learning — atenção, FFN, projeções, tudo vira GEMM.

É o "motor" da GPU: na A100, GEMM atinge 312 TFLOPS enquanto operações não-matmul (exponencial, divisão) atingem só 19.5 TFLOPS — 16× mais lentas. Otimizar atenção é essencialmente reduzir operações não-GEMM.

🔄 Auto-regressivo

Modelo que gera saída um token por vez, e cada token gerado é adicionado à entrada para gerar o próximo. O GPT é auto-regressivo: gera palavra 1, usa palavra 1 para gerar 2, usa 1 e 2 para gerar 3, etc.

Intuição: é como escrever um texto — cada palavra que você escreve depende de todas as palavras que já escreveu antes. Você não decide a palavra 50 antes de ter escrito as 49 anteriores.

🔍 Beam Search

Algoritmo de busca na inferência (geração de texto) que mantém as k melhores continuações parciais, em vez de escolher sempre a melhor próxima palavra (greedy).

Intuição (beam=4): em vez de sempre escolher A MAIS provável próxima palavra (que pode levar a um beco sem saída), o modelo mantém 4 "hipóteses" simultâneas e no final escolhe a frase completa mais provável. No Transformer original, beam size = 4 com penalidade de comprimento α = 0.6.

🎲 Dropout

Técnica de regularização que, durante o treino, zera aleatoriamente uma fração p dos neurônios (p=0.1 → 10% desligados). Força a rede a não depender de neurônios específicos, melhorando a generalização.

Intuição: estudar para uma prova em grupo onde a cada dia alguns colegas faltam. Você aprende a não depender de ninguém específico. Na prova (inferência), todo mundo comparece.

No Transformer: dropout aplicado após cada sub-camada, nos embeddings, e nos pesos de atenção. p=0.1 no paper original.

⚡ ReLU (Rectified Linear Unit)

Função de ativação simples: ReLU(x) = max(0, x). Se x é negativo, vira 0; se positivo, passa direto.

Intuição: um interruptor — ou o neurônio "dispara" (valor positivo) ou fica "desligado" (negativo vira zero). Simples mas a não-linearidade que permite redes profundas aprenderem funções complexas.

No Transformer: usada na FFN: FFN(x) = ReLU(xW₁ + b₁)W₂ + b₂. Modelos modernos (GPT, LLaMA) usam GELU ou SwiGLU.

🤝 Ensemble

Combinação de múltiplos modelos treinados independentemente para melhorar a qualidade final. As previsões são combinadas por média ou votação.

Intuição: pergunte a 5 especialistas e combine as respostas — erros individuais tendem a se cancelar. No Transformer (2017): o modelo BASE único já superava ensembles de RNNs. O modelo BIG com ensemble batia todo mundo.

🏷️ Label Smoothing

Técnica que "suaviza" os rótulos de treinamento. Em vez de dizer "classe correta = 100%, erradas = 0%", distribui um pouco de probabilidade para as classes erradas.

Com ε=0.1 (Transformer): classe correta = 90%, cada classe errada = 0.1/(|V|-1).

Intuição: um professor que aceita que às vezes há mais de uma resposta razoável, em vez de dizer "só essa é a certa". Evita overfitting (decorar em vez de aprender).

Curiosidade: label smoothing piora perplexidade (modelo fica menos "certo") mas melhora BLEU (traduções melhores). Certeza absoluta nem sempre é o melhor.
