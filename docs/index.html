1. Tokenização — Byte-Pair Encoding & Subword Tokenization

🟢 Dominado

O Problema

Como representar texto para uma rede neural de forma eficiente, sem vocabulários gigantes e sem perder palavras raras ou desconhecidas?

Contexto Histórico

  • Word-level: cada palavra = um ID. Vocabulário enorme, palavras desconhecidas viram <UNK>
  • Character-level: cada caractere = um ID. Vocabulário pequeno, mas sequências ficam longas demais
  • Solução (2016): Subword tokenization (BPE de Sennrich et al.) — meio-termo que quebra palavras em pedaços frequentes

Intuição Central

Se você conhece os pedaços "gat" + "inho", entende "gatinho" mesmo sem nunca ter visto essa palavra. O BPE extrai esses pedaços automaticamente do corpus.

Analogia do LEGO: word-level são peças únicas rígidas, character-level são pinos minúsculos, subword são peças de tamanho certo que se encaixam para formar qualquer palavra.

Como Funciona

  1. Começa com todas as palavras quebradas em caracteres
  2. Conta todos os pares de tokens adjacentes no corpus
  3. Funde o par mais frequente em um novo token
  4. Repete até atingir o tamanho de vocabulário desejado (ex: 37.000)

Exemplo: "gato gato gatos" → g a t o → funde "ga" → funde "gat" → vocabulário final inclui "gat", permitindo tokenizar "gatos" como gat o s.

Impacto Prático

ModeloMétodoVocabulário
Transformer (EN→DE)BPE~37.000 tokens
Transformer (EN→FR)Word-Piece~32.000 tokens
GPT-3BPE~50.000 tokens
LLaMABPE~32.000 tokens

Tokenização é pré-processamento em CPU, não afeta tempo de GPU.

Conexões

  • Embeddings: cada token (ID inteiro) é convertido em vetor denso
  • Vocabulário × d_model: define o tamanho da matriz de embedding
  • Weight tying: Transformer compartilha pesos entre embedding de entrada e saída

Perguntas de Revisão

  1. Por que não usamos palavras inteiras como tokens?
  2. Qual a diferença entre tokenização word-level, character-level e subword?
  3. Explique o algoritmo BPE com suas próprias palavras
  4. Por que "gatinho" e "gatos" compartilham tokens no BPE?
  5. O que acontece se o vocabulário for muito pequeno? E muito grande?

2. Embeddings — De Inteiros a Vetores com Significado

🟢 Dominado

O Problema

Como representar tokens (inteiros) como vetores densos que capturem significado semântico, permitindo que palavras similares tenham representações similares?

Contexto Histórico

  • One-hot encoding (pré-2013): vocabulário de 37K = vetores de 37K dimensões com um único 1. Distâncias sem sentido entre palavras
  • Word2Vec (Mikolov et al., 2013): vetores densos de ~300 dimensões capturam relações como rei - homem + mulher ≈ rainha
  • Transformer (2017): embeddings aprendidos de 512 dimensões, com weight tying e fator de escala √d_model

Intuição Central

Em vez de listar todas as palavras do mundo (one-hot), descreva uma palavra por características abstratas: tom emocional, tipo gramatical, domínio semântico. 512 dimensões bastam para capturar esses aspectos.

Analogia: Ficha de biblioteca com 512 campos (gênero, tema, época, dificuldade...) vs. uma prateleira exclusiva por livro.

Matemática

  • Matriz de embedding: E ∈ ℝ^{|V| × d_model}
  • Lookup: embedding(token) = E[t_id] × √d_model
  • Inicialização: E_ij ~ N(0, 1/d_model)
  • Weight tying: mesma E usada no encoder, decoder e projeção pré-softmax

Impacto Prático

  • Parâmetros: 37K × 512 = ~19M floats ≈ 76 MB (float32)
  • GPT-3: 50K × 12.288 = 614M parâmetros ≈ 2.5 GB só nos embeddings
  • Lookup é O(1) por token — extremamente rápida na GPU

Conexões

  • Positional Encoding: soma-se ao embedding para injetar posição
  • Self-Attention: opera sobre os vetores de embedding
  • Weight tying: reduz parâmetros e força consistência entrada-saída

Perguntas de Revisão

  1. O que é um embedding e por que ele é melhor que one-hot encoding?
  2. Por que o Transformer multiplica os embeddings por √d_model?
  3. O que é weight tying e por que ele é usado?
  4. Quantos parâmetros tem a matriz de embedding para |V|=37K e d_model=512?
  5. Por que a inicialização dos embeddings importa?

3. Self-Attention — O Coração do Transformer

🟢 Dominado

O Problema

Como permitir que cada token de uma sequência "olhe" diretamente para todos os outros tokens em O(1) passos sequenciais, sem a degradação de informação das RNNs?

Contexto Histórico

  • RNN (pré-2017): cada token via apenas o estado oculto do anterior. Informação se degradava. Impossível paralelizar
  • Atenção de Bahdanau (2014): atenção entre encoder e decoder, mas ainda com RNN processando sequencialmente
  • Transformer (2017): remove RNNs completamente. Cada token atende a todos os outros em O(1) passos

Intuição Central

Imagine uma sala de aula onde cada aluno (token) tem uma Query ("o que eu quero saber?"), uma Key ("o que eu ofereço?") e um Value ("qual é minha informação?"). Cada aluno compara sua Query com as Keys de todos, e faz uma média ponderada dos Values de quem for mais compatível.

Analogia: Busca em banco de dados — Query é sua pergunta, Key é o índice, Value é o conteúdo retornado. Mas aqui tudo é "soft" (ponderado e diferenciável).

Matemática

A equação completa:

Scaled Dot-Product Attention 
Attention(Q, K, V) = softmax(Q·Kᵀ / √d_k)·V

Onde:
  Q = X·W_Q   (n × d_k)
  K = X·W_K   (n × d_k)
  V = X·W_V   (n × d_v)
  d_k = 64 (para h=8 cabeças)

O fator 1/√d_k é crucial: sem ele, produtos escalares grandes saturam o softmax, matando o gradiente. Para d_k=64, divide-se por 8.

Impacto Prático

Seq LenMatriz de AtençãoMemória (float32)
512262K elementos~1 MB
20484.2M elementos~17 MB
819267M elementos~268 MB
32K1B elementos~4 GB

Complexidade O(n²·d). O produto Q·Kᵀ é o gargalo. Flash Attention resolve isso em memória.

Conexões

  • Multi-Head Attention: 8 atenções em paralelo
  • Masked Self-Attention: decoder usa máscara para não ver tokens futuros
  • Cross-Attention: decoder atende à saída do encoder
  • Flash Attention: otimização de memória para O(n²)

Perguntas de Revisão

  1. O que significam Query, Key e Value? Dê a analogia com suas palavras
  2. Por que dividimos QKᵀ por √d_k? O que acontece se não dividirmos?
  3. Por que a complexidade é O(n²·d)? Qual parte domina?
  4. O que significa um peso de atenção 0.95 na posição (i=3, j=7)?
  5. Por que self-attention captura dependências longas melhor que RNN?

4. Multi-Head Attention — 8 Atenções em Paralelo

🟢 Dominado

O Problema

Uma única função de atenção precisa capturar relações sintáticas, semânticas, posicionais e de correferência simultaneamente nos mesmos 64 números — uma "média" que perde nuances.

Contexto Histórico

  • Atenção de Bahdanau (2014) e Luong (2015): uma única função de atenção
  • Transformer (2017): propõe h=8 atenções em paralelo, cada uma com seus W_Q, W_K, W_V
  • Linha A (Tabela 3): cabeça única perde 0.9 BLEU; cabeças demais também pioram → ponto ótimo em h=8

Intuição Central

Em vez de UM perito analisando a cena do crime, chame 8 especialistas: um de digitais, um de balística, um de DNA, um de documentos... Cada um olha a mesma cena com ferramentas diferentes. Depois, junte todos os laudos.

Analogia: Detetives especializados — cada cabeça aprende um "tipo de pergunta" diferente sobre a sequência.

Matemática

Multi-Head Attention 
MultiHead(Q, K, V) = Concat(head₁, ..., head_h)·W_O
head_i = Attention(Q·W_Q_i, K·W_K_i, V·W_V_i)

  h = 8 cabeças
  d_k = d_v = d_model/h = 512/8 = 64
  W_Q_i, W_K_i ∈ ℝ^{512×64}, W_V_i ∈ ℝ^{512×64}
  W_O ∈ ℝ^{512×512}

O custo total é similar a 1 cabeça com dimensionalidade total: 8 × O(n²×64) = O(n²×512).

Impacto Prático

  • Cabeças descobrem padrões interpretáveis sozinhas: sintaxe, correferência, adjacência
  • Exemplo real: "its" atende fortemente a "The Law" — correferência sem supervisão
  • Dropout (p=0.1) nos pesos de atenção evita dependência excessiva de uma cabeça

Conexões

  • Masked Multi-Head Attention: decoder usa máscara para impedir atenção ao futuro
  • Cross-Attention: decoder atende à saída do encoder
  • MQA / GQA: variações modernas que compartilham K e V entre cabeças

Perguntas de Revisão

  1. Por que usar 8 cabeças em vez de 1?
  2. Explique por que o custo de 8 cabeças (d_k=64) é similar a 1 cabeça (d_k=512)
  3. Qual o papel da matriz W_O?
  4. O que uma cabeça com atenção sempre uniforme indicaria?
  5. Por que cabeça única e cabeças demais são ambas piores que h=8?

5. Os 3 Tipos de Atenção no Transformer

🟢 Dominado

O Problema

O Transformer precisa de diferentes "regras de visibilidade" para diferentes tarefas: entender a entrada (tudo visível), gerar saída (só passado visível) e conectar entrada-saída (decoder consulta encoder).

Intuição Central

Um tradutor humano usa 3 operações mentais:

  1. Ler a frase original: conecta todas as palavras entre si para entender o significado completo
  2. Escrever a tradução: cada palavra nova só considera o que já foi escrito
  3. Consultar o original: enquanto escreve, revisita a frase fonte para garantir fidelidade

Os 3 Tipos

TipoQ, K, V vêm deMáscara?Custo
Encoder Self-AttentionTodos do encoderNãoO(n_enc²)
Decoder Masked Self-AttentionTodos do decoderTriangularO(n_dec²)
Cross-AttentionQ=decoder, K,V=encoderNãoO(n_enc × n_dec)

Matemática

Encoder Self-Attention: Attention(X_enc·W_Q, X_enc·W_K, X_enc·W_V)

Decoder Masked: Attention(X_dec·W_Q, X_dec·W_K, X_dec·W_V + M) onde M tem −∞ nas posições futuras

Cross-Attention: Attention(X_dec·W_Q, Z_enc·W_K, Z_enc·W_V)

Conexões

  • GPT: decoder-only (apenas masked self-attention)
  • BERT: encoder-only (apenas self-attention sem máscara)
  • KV-cache: otimização de inferência que armazena K e V no decoder

Perguntas de Revisão

  1. Por que o decoder precisa de máscara na self-attention?
  2. Qual a diferença entre self-attention e cross-attention?
  3. Por que a cross-attention NÃO tem máscara?
  4. GPT (decoder-only) usa quais tipos de atenção? E BERT (encoder-only)?
  5. O que aconteceria se a cross-attention usasse Q do encoder em vez do decoder?

6. Positional Encoding — Como o Modelo Sabe a Ordem?

🟢 Dominado

O Problema

A self-attention trata a sequência como um conjunto não-ordenado. Sem informação de posição, "A ama B" e "B ama A" produzem representações idênticas.

Contexto Histórico

  • RNNs: posição implícita no processamento sequencial
  • CNNs (ConvS2S, ByteNet): posição implícita via janela de convolução
  • Transformer (2017): precisa injetar posição explicitamente, já que removeu RNNs e CNNs

Intuição Central

Analogia do relógio: ponteiro dos segundos (alta frequência), minutos (média), horas (baixa). A combinação única das posições dos 3 ponteiros codifica qualquer horário. O PE faz o mesmo com 256 "ponteiros" (pares seno/cosseno).

Matemática

Positional Encoding — Senos e Cossenos 
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

  pos = 0, 1, 2, ..., n-1
  i   = 0, 1, ..., d_model/2 - 1
  Valores sempre em [-1, 1]

Frequências formam progressão geométrica de 2π a ~10000×2π. PE(pos+k) é função linear de PE(pos) → facilita aprendizado de posições relativas.

Impacto Prático

  • PE senoidal vs. aprendido: resultados quase idênticos (Tabela 3, linha E)
  • Modelos modernos (GPT, LLaMA): preferem PE aprendido ou RoPE
  • Sem √d_model nos embeddings, o PE (±1) dominaria o sinal semântico

Conexões

  • RoPE (Rotary Position Embedding): evolução usada em LLaMA, Mistral
  • ALiBi: alternativa sem PE explícito — viés linear na atenção

Perguntas de Revisão

  1. Por que o Transformer precisa de positional encoding?
  2. Por que usar senos/cossenos de diferentes frequências?
  3. Por que somar o PE ao embedding e não concatenar?
  4. O que significa que PE(pos+k) é função linear de PE(pos)?
  5. Por que as frequências formam progressão geométrica?

7. Feed-Forward, Conexões Residuais e LayerNorm

🟢 Dominado

O Problema

Como empilhar 6+6 camadas sem sofrer com gradiente evanescente, e como cada token processa individualmente a informação após a atenção?

Contexto Histórico

  • ResNet (He et al., 2015): introduz conexões residuais (skip connections) — F(x) + x
  • BatchNorm (Ioffe & Szegedy, 2015): normaliza ativações, mas depende do batch
  • LayerNorm (Ba et al., 2016): normaliza por token, ideal para sequências

Intuição Central

Feed-Forward: Após a reunião de equipe (atenção), cada token processa as ideias sozinho. Expande 512→2048 (rascunho), comprime 2048→512 (resumo final).

Conexão Residual: Em vez de reescrever do zero, faça pequenas edições: saída = SubLayer(x) + x. O "atalho" permite que o sinal pule a transformação.

LayerNorm: Equalizador de áudio — ajusta cada token para média 0 e desvio padrão 1. Estabiliza o treino.

Matemática

FFN: FFN(x) = ReLU(x·W₁ + b₁)·W₂ + b₂ — W₁: 512×2048, W₂: 2048×512

LayerNorm: LayerNorm(x) = γ · (x - μ)/σ + β — μ e σ por token, γ e β aprendidos

ComponenteParâmetros
Multi-Head Attention (4 matrizes)~1.05M
FFN (2 camadas)~2.1M
2 LayerNorms~2K
Total por camada~3.15M

Modelo base (12 camadas) + embeddings ≈ ~65M parâmetros

Conexões

  • Pré-norm vs. pós-norm: modelos modernos usam LayerNorm ANTES da SubLayer
  • SwiGLU: variante moderna da FFN com ativação diferente
  • Gradient Checkpointing: explora camadas residuais para economizar memória

Perguntas de Revisão

  1. Qual a função da conexão residual?
  2. Por que a FFN expande para 2048 e depois contrai?
  3. Qual a diferença entre BatchNorm e LayerNorm?
  4. Quantos parâmetros tem o modelo base (~65M)?
  5. Por que modelos modernos preferem pré-norm a pós-norm?

8. Transformer — Arquitetura Completa e Treinamento

🟢 Dominado

O Problema

Como substituir completamente RNNs e CNNs por um mecanismo puramente baseado em atenção, com melhor qualidade e menos tempo de treino?

Contexto Histórico

Pré-2017: estado da arte em tradução usava ensembles de RNNs. Melhor EN→DE: ~26 BLEU. O Transformer (2017) atingiu 28.4 BLEU com modelo único, treinado em 3.5 dias com 8 GPUs P100.

Arquitetura Completa

Pipeline do Transformer 
ENTRADA → Tokenização → Embedding + PE → ENCODER (6×) → DECODER (6×) → Linear + Softmax → SAÍDA

Encoder: Self-Attention (8 cabeças) + FFN (512→2048→512)
Decoder: Masked Self-Attn + Cross-Attn + FFN
Ambos: residual + LayerNorm (pós-norm)

Hiperparâmetros (Modelo Base)

ParâmetroValorParâmetroValor
d_model512P_drop0.1
d_ff2048Label smoothing0.1
h (cabeças)8Batch~25K tokens
d_k, d_v64Warmup4000 steps
N (camadas)6 enc + 6 decTraining steps100K

Treinamento

  • Optimizer: Adam (β₁=0.9, β₂=0.98, ε=10⁻⁹)
  • Learning rate: warmup linear por 4000 steps, depois decaimento ∝ 1/√step
  • Regularização: Dropout 0.1 + Label Smoothing 0.1
  • Inferência: Beam search com tamanho 4, penalidade de comprimento α=0.6

Resultados

Conexões

  • Flash Attention: resolve O(n²) em memória na atenção
  • LoRA: fine-tuning eficiente congelando pesos do Transformer
  • Quantização: reduz precisão dos pesos para economia de memória
  • Paralelismo: distribui treino entre múltiplas GPUs

Técnicas de Eficiência de Atenção

Flash Attention — Atenção Exata com O(N) de Memória

🟢 Dominado

O Problema

Dados Q, K, V ∈ ℝN×d, a atenção padrão calcula:

Atenção Padrão — 3 passos, 2 escritas N×N na HBM 
S = Q @ K.T          // [N×N] → escreve na HBM
P = softmax(S)        // [N×N] → escreve na HBM
O = P @ V             // [N×d] → escreve na HBM

Total de acessos HBM: Θ(Nd + N²)

Para N=1024, d=64, 16 cabeças, batch=64 em A100: 40.3 GB de leitura/escrita na HBM. O Flash Attention reduz isso para 4.4 GB — 9× menos — eliminando as escritas das matrizes S e P. A GPU não é lenta calculando; é lenta movendo dados.

Comprimento NMatriz N×N (FP16)Economia FA
1,0242 MB10×
4,09634 MB20×
32,7682.1 GB~50×
65,5368.6 GB~100×

Contexto Histórico

Antes de 2022, a comunidade tentava resolver O(N²) com aproximações: sparse attention, Linformer (projeção low-rank), Performer (FAVOR+). Todas perdiam qualidade. O insight veio de Rabe & Staats (2021): "Self-Attention Does Not Need O(n²) Memory" — provaram que tiling funciona para atenção, mas sem implementação IO-aware eficiente. O Flash Attention pegou essa ideia e construiu um kernel CUDA que atinge o limite inferior teórico de acessos à memória.

Intuição Central

A GPU tem dois níveis de memória radicalmente diferentes:

MemóriaCapacidade (A100)BandaLatência
HBM (VRAM)40-80 GB1.5-2.0 TB/s~300 ciclos
SRAM (shared mem)192 KB / SM~19 TB/s~20-30 ciclos

O algoritmo ingênuo trata a GPU como se só existisse HBM. O Flash Attention usa a SRAM como cache explícito: divide as matrizes em tiles que cabem na SRAM, processa cada tile completamente e nunca escreve a matriz N×N na HBM.

📄 Paper 1: FlashAttention (Dao et al., NeurIPS 2022 — Best Paper)

Algoritmo 1 — Forward Pass Completo

Tamanhos de bloco (derivados do tamanho M da SRAM): Bc = ⌈M/(4d)⌉ para K,V; Br = min(⌈M/(4d)⌉, d) para Q,O.

Forward FA1 — Pseudocódigo extraído do paper (Algorithm 1) 
Entrada: Q, K, V ∈ ℝ^(N×d) na HBM, blocos B_c, B_r

1.  Divide Q em T_r = ⌈N/B_r⌉ blocos Q_1...Q_{T_r} de tamanho B_r×d
2.  Divide K, V em T_c = ⌈N/B_c⌉ blocos K_1...K_{T_c}, V_1...V_{T_c}
3.  Inicializa O = (0)_{N×d}, ℓ = (0)_N, m = (-∞)_N na HBM

4.  para j = 1 até T_c:                     // outer loop: K,V blocks
5.      carrega K_j, V_j da HBM → SRAM
6.      para i = 1 até T_r:                 // inner loop: Q blocks
7.          carrega Q_i, O_i, ℓ_i, m_i → SRAM
8.
9.          // Passo 1: scores de atenção
10.         S_ij = τ · Q_i @ K_j^T           // [B_r × B_c] — PEQUENO!
11.         S_ij = MASK(S_ij)                // causal/padding mask
12.
13.         // Passo 2: online softmax
14.         m̃_ij = rowmax(S_ij)              // [B_r,]
15.         P̃_ij = exp(S_ij - m̃_ij)           // [B_r × B_c] pointwise
16.         ℓ̃_ij = rowsum(P̃_ij)              // [B_r,]
17.
18.         // Passo 3: atualiza estatísticas (CORREÇÃO CRÍTICA!)
19.         m_novo = max(m_i, m̃_ij)
20.         ℓ_novo = e^(m_i-m_novo)·ℓ_i + e^(m̃_ij-m_novo)·ℓ̃_ij
21.
22.         // Passo 4: dropout (se aplicável)
23.         P̃_ij = dropout(P̃_ij, p_drop)
24.
25.         // Passo 5: acumula output com correção de escala
26.         O_i = diag(ℓ_novo)⁻¹ · (diag(ℓ_i)·e^(m_i-m_novo)·O_i
27.                                   + e^(m̃_ij-m_novo)·P̃_ij @ V_j)
28.
29.         escreve O_i, ℓ_i←ℓ_novo, m_i←m_novo → HBM
30.     fim para
31. fim para
32. retorna O, ℓ, m, R

O truque do online softmax: a correção e^(m_velho - m_novo) rescala todos os exponenciais acumulados para a base do novo máximo. Quando um bloco novo tem máximo maior que o anterior, os exponenciais antigos precisam ser "encolhidos". Sem isso: overflow se o novo máximo for maior, underflow se for menor. Backward Pass — Recomputation

O backward não salva S e P (N×N cada). Recalcula cada bloco S_ij e P_ij na SRAM usando Q_i, K_j e as estatísticas armazenadas.

📄 Paper 2: FlashAttention-2 (Dao, 2023)

Por que o FA1 era ineficiente (25-40% do pico na A100)

  1. Split-K no particionamento de warps: K e V divididos entre warps, Q compartilhado. Cada warp escrevia seu resultado parcial na shared memory → sincronização → soma. Essas leituras/escritas extras dominavam o tempo.
  2. FLOPs não-matmul excessivos: cada FLOP não-matmul é 16× mais caro que matmul na A100 (312 TFLOPS matmul FP16 vs 19.5 TFLOPS não-matmul FP32).
  3. Baixa ocupação com sequências longas: paralelizava só sobre batch×heads, não sobre sequence length. Com seqlen grande → batch pequeno → GPUs subutilizadas.

Três melhorias do FA2

1. Redução de FLOPs não-matmul:

  • Tweak A: output mantido "não-escalado" até o final do loop. Só normaliza uma vez no fim: O = diag(ℓ)^{-1} · Õ. Elimina uma multiplicação por diag por iteração.
  • Tweak B: em vez de salvar m e ℓ separados, salva L = m + log(ℓ) (logsumexp). Backward usa L diretamente.

📄 Paper 3: FlashAttention-3 (Dao et al., 2024) — ⚡ Este é o paper para implementar hoje

Por que FA2 atingia só 35% na H100

  1. mma.sync bloqueante: instrução da Ampere que bloqueia a warp. Na H100, só atinge ~2/3 do throughput máximo dos Tensor Cores.
  2. Sem TMA: transferências HBM↔SRAM eram manuais (load/store), consumindo registradores e sem sobreposição com computação.
  3. Sem FP8: throughput FP8 é 2× FP16 (1978 vs 989 TFLOPS na H100 SXM5).
  4. Gargalo da exponencial: unidade de funções especiais tem apenas 3.9 TFLOPS na H100 — 256× menos que matmul FP16. Para head dim 128, exp pode consumir 50% do tempo. Com FP8 seria pior ainda (matmul dobra, exp constante).

Três inovações do FA3

A. WGMMA (Warp Group MMA) + Warp Specialization:

  • WGMMA: instrução assíncrona exclusiva da Hopper. Substitui mma.sync. O warp group (4 warps) dispara a multiplicação e continua executando enquanto ela roda nos Tensor Cores.
  • Inter-Warpgroup Pingpong: dois warpgroups (A: GEMMs, B: softmax) alternam via bar.sync. A softmax roda "na sombra" dos GEMMs do outro warpgroup.
  • Intra-Warpgroup Pipelining: pipeline de 2 estágios com acumuladores separados. Partes do softmax executam enquanto os GEMMs estão em voo. Tradeoff: maior pressão de registradores.

B. TMA (Tensor Memory Accelerator): Hardware dedicado para transferências HBM↔SRAM. Cuida de cálculo de índices e predicação de bounds. Libera registradores. Assíncrono — sobrepõe carga de dados com computação.

C. FP8 com Incoherent Processing:

  • Multiplica Q e K por matriz de Hadamard aleatória — "espalha" outliers para que a energia fique uniforme
  • Custo: O(d log d) por cabeça, fundido com rotary embeddings (praticamente grátis)
  • Erro de quantização 2.6× menor que FP8 baseline
  • Forward inteiro em FP8 (~2× throughput); backward em BF16 para estabilidade

🏭 HBM vs SRAM — Hierarquia de Memória da GPU

HBM (High Bandwidth Memory): memória principal da GPU (VRAM). Grande (40-80 GB) mas lenta (latência de centenas de ciclos). Todas as matrizes do modelo ficam aqui. Analogia: um galpão do outro lado da rua — cabe tudo, mas cada viagem é demorada.

SRAM (Static RAM / Shared Memory): memória dentro de cada Streaming Multiprocessor. Pequena (~228 KB por SM na H100) mas muito rápida (~25 ciclos de latência). Analogia: sua mesa de trabalho — cabe pouco, mas tudo ao alcance da mão.

Por que importa no Flash Attention: TODO o algoritmo gira em torno de nunca escrever a matriz N×N na HBM. Em vez disso, processa blocos pequenos na SRAM. Cada acesso evitado à HBM é ganho direto de velocidade.

🧩 Sparse Attention

Família de técnicas que calcula apenas um subconjunto das conexões entre tokens, em vez de todas as N×N conexões. É como ler um livro e só consultar palavras do mesmo parágrafo, em vez de comparar cada palavra com todas as outras.

Flash Attention NÃO é sparse attention — ele calcula a atenção exata (todas as conexões). Mas antes de 2022, sparse attention era a principal estratégia para reduzir o custo O(N²) da atenção.

📐 Linformer

Uma das primeiras tentativas de reduzir o custo da atenção usando projeção de baixo rank. A ideia: a matriz de atenção N×N pode ser aproximada por duas matrizes menores N×k e k×N, onde k é pequeno (~256). Complexidade: O(N²) → O(N·k).

Limitação: não escala para sequências mais longas que as do treino. Se treinar com N=512 e inferir com N=4096, degrada.

Relação com Flash Attention: o Linformer mostrou que a matriz de atenção tem baixo rank (muita redundância). Mas é uma aproximação — o Flash Attention veio depois e mostrou que dava para ser exato e eficiente.

⬆ Voltar ao Flash Attention

🎭 Performer

Usa um truque matemático chamado FAVOR+ (Fast Attention Via Orthogonal Random features) para aproximar o softmax com kernels e projeções aleatórias. Complexidade linear O(N) — foi o primeiro mecanismo de atenção linear que funcionou bem.

Performer vs Flash Attention: atacam o mesmo problema de ângulos opostos. Performer muda a matemática (aproximação do softmax → O(N) computação). Flash Attention muda a engenharia (IO-awareness → O(N) memória, computação ainda O(N²)). Para qualidade crítica, FA venceu.

🧱 Tiles (Ladrilhamento)

Técnica onde uma matriz grande é dividida em blocos menores que cabem na memória rápida (SRAM). Cada bloco é carregado, processado completamente e descartado antes do próximo.

Analogia: pintar uma parede de 10m×10m — você pinta um quadrado de 1m×1m de cada vez. A parede inteira nunca está na sua mão, só o ladrilho atual.

A matriz de atenção N×N é dividida em tiles de tamanho Br×Bc (ex: 128×128):

Matriz N×N dividida em tiles 
┌─────────┬─────────┬─────────┐
│ Tile    │ Tile    │ Tile    │
│ [0,0]   │ [0,1]   │ [0,2]   │  ← 128×128 cada
├─────────┼─────────┼─────────┤
│ Tile    │ Tile    │ Tile    │
│ [1,0]   │ [1,1]   │ [1,2]   │
├─────────┼─────────┼─────────┤
│ Tile    │ Tile    │ Tile    │
│ [2,0]   │ [2,1]   │ [2,2]   │
└─────────┴─────────┴─────────┘

Cada tile de 128×128 em float16 ocupa 32 KB — cabe na SRAM de 228 KB.

⚡ Warp e WGMMA (CUDA)

Warp: grupo de 32 threads que executam juntas a mesma instrução sobre dados diferentes (modelo SIMT). É a unidade fundamental de execução em GPUs NVIDIA. Uma H100 gerencia 64 warps simultaneamente por SM.

WGMMA (Warp Group Matrix Multiply-Accumulate): instrução exclusiva da arquitetura Hopper (H100) que permite que um grupo de warps dispare uma multiplicação de matrizes de forma assíncrona — o warp não espera o resultado, continua executando enquanto a multiplicação roda nos Tensor Cores.

🔢 Online Softmax

Algoritmo que calcula o softmax incrementalmente, processando pedaços do vetor sem nunca ver todos os valores de uma vez. Mantém duas variáveis atualizadas bloco a bloco:

  • m: máximo corrente (para estabilidade numérica)
  • l: soma corrente dos exponenciais (para normalização)

Quando um novo bloco chega, a correção exp(m_velho - m_novo) rescala todos os exponenciais antigos para a nova base. Sem isso, os números estouram (overflow) ou viram zero (underflow).

Exemplo numérico simples 
Bloco 1: scores = [2, 1, 3]
  m = 3, l = e^(2-3) + e^(1-3) + e^(3-3) = 1.503

Bloco 2: scores = [5, 0, 4]  ← novo máximo 5 > 3!
  m_novo = 5
  l = 1.503 * e^(3-5) + e^(5-5) + e^(0-5) + e^(4-5)
    = 1.503 * 0.135 + 1 + 0.007 + 0.368
    = 1.578

O fator e^(3-5) = 0.135 "encolheu" os exponenciais do bloco 1 para a base correta.

Por que importa: é o que viabiliza o tiling no cálculo da atenção. Sem o online softmax, você precisaria da matriz N×N inteira — exatamente o que o Flash Attention evita.
